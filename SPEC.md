# ТЗ: Управление мышью ПК с телефона через гироскоп

## 1. Цель и общая идея

Веб-приложение, превращающее смартфон в указку/пульт для управления курсором мыши на ПК.
Телефон отдаёт данные ориентации (гироскоп/акселерометр/компас) браузера, ПК-сервер
принимает их по WebSocket и двигает системный курсор. Управление — «как лазерной указкой»:
наводишь телефон — курсор едет за наведением.

Проект состоит **строго из двух файлов**:

1. **Клиент** — один HTML-файл (вся разметка + стили + JS внутри), открывается в браузере телефона.
2. **Сервер/демон** — на **Rust**, собирается в один статический бинарь, запускается на ПК, которым
   управляют. HTML-клиент встраивается прямо в бинарь, так что на ПК поставляется фактически один файл.

Цель сервера: дешёвый кроссплатформенный фоновый демон без внешних рантаймов и без системного OpenSSL.
Клиент — без сборок/бандлеров/npm; стили можно тянуть с CDN (Tailwind), не обязательно.
Стек и крейты Rust — в разделе 13.

## 2. Архитектура и сетевое взаимодействие

### 2.1. Один порт на всё
Сервер слушает **один TCP-порт** (по умолчанию 8081) и обслуживает на нём И веб-страницу (HTTP/HTTPS),
И WebSocket. Логика разделения по входящему запросу:

- если в запросе есть заголовок `Upgrade: websocket` → это WebSocket-рукопожатие, пропускаем дальше в WS-обработчик;
- иначе это обычный браузер грузит страницу → отдаём HTML как HTTP-ответ.

Смысл: телефон один раз принимает самоподписанный сертификат, и страница, и сокет работают по одному адресу/порту.

### 2.2. HTTPS/WSS обязателен для iOS
iOS (Safari) выдаёт доступ к датчикам ориентации **только в защищённом контексте** (HTTPS) и **только по жесту пользователя**.
Поэтому:

- сервер при старте генерирует **самоподписанный сертификат** средствами чистого Rust (крейт `rcgen`,
  без системного OpenSSL), в SAN кладёт выбранный LAN-IP, плюс `127.0.0.1` и `localhost`, срок ~10 лет;
- сертификат кэшируется на диске; если LAN-IP сменился — перегенерируется (запоминаем, для какого IP он выпущен);
- TLS терминируется через `rustls`; при невозможности поднять TLS — откат на чистый HTTP с предупреждением,
  что iPhone не даст датчики.

Android (Chrome) обычно работает и по HTTP, и без явного запроса разрешений, но единый HTTPS-путь упрощает жизнь.

### 2.3. Автоопределение LAN-IP
Сервер должен подсказать пользователю правильный адрес для телефона:

- перечислить все приватные IPv4 адреса хоста;
- **не** доверять «интерфейсу маршрута по умолчанию» (при активном VPN это будет VPN-адрес);
- ранжировать по диапазону: `192.168.*` приоритетнее `172.*` приоритетнее `10.*`; диапазон VirtualBox
  host-only `192.168.56.*` штрафовать;
- дать возможность жёстко задать предпочитаемый IP константой (override);
- лучший кандидат помечается как **рекомендуемый** и предлагается по умолчанию в интерактивном
  выборе при старте (см. раздел 12).

### 2.4. Протокол сообщений
Все сообщения — JSON, направление **только телефон → сервер** (сервер ничего не шлёт обратно;
статус соединения телефон выводит сам по событиям open/close сокета).

Типы сообщений (по смыслу, состав полей — на усмотрение реализации):

| Сообщение | Назначение | Полезная нагрузка |
|---|---|---|
| смена режима | переключение absolute/relative | режим |
| калибровка точки | сохранить ориентацию для угла экрана | идентификатор точки (TL/TR/BL/BR/Center), углы beta и alpha |
| сброс калибровки | очистить точки | — |
| движение | основной поток данных ориентации | beta, gamma, alpha; опционально ускорение по 3 осям |
| нажатие кнопки | прижать ЛКМ/ПКМ | какая кнопка |
| отпускание кнопки | отпустить ЛКМ/ПКМ | какая кнопка |
| скролл | прокрутка колеса | величина дельты |
| вкл/выкл анти-джиттер | переключатель | bool |
| чувствительность сглаживания | значение слайдера | число |

Поток «движения» идёт часто (~60 раз/сек, привязан к requestAnimationFrame), поэтому
логирование на сервере должно быть **тротлингованным** (не чаще, например, раза в 0.25с на категорию).

### 2.5. Зависимости
- Сервер (Rust): async-рантайм + WebSocket/HTTP, TLS, генерация серта, управление мышью, геометрия
  мониторов, TUI и QR — конкретные крейты перечислены в разделе 13.
- Деградация должна быть мягкой: не удалось захватить устройство ввода → не двигаем курсор, но не падаем;
  не удалось получить раскладку мониторов → откат на один экран 1920×1080; не удалось поднять TLS → откат на HTTP.

## 3. Датчики телефона (что приходит из браузера)

- **deviceorientation**: `alpha` (0–360, азимут/компас, рыскание вокруг вертикали), `beta` (тангаж,
  наклон вперёд-назад, ~ −180..180), `gamma` (крен, наклон вбок, ~ −90..90).
- **devicemotion**: `accelerationIncludingGravity` (ускорение с гравитацией по 3 осям) — используется
  только для анти-джиттера.

Важно: разрешение на датчики на iOS запрашивается **синхронно внутри обработчика тапа** (кнопка Connect),
иначе доступ не выдаётся.

## 4. Экраны/состояния клиента (UI)

1. **Setup** — поле ввода IP сервера (предзаполняется из hostname страницы) и кнопка Connect.
   По тапу: запрос разрешений на датчики → подключение к сокету.
2. **Mode selection** — выбор режима:
   - «Air Mouse / лазерная указка» (relative) — без калибровки, рекомендуемый, основной;
   - «Absolute» — точная привязка к экрану, требует калибровки;
   - переключатель «динамический анти-джиттер»;
   - слайдер «сглаживание».
3. **Calibration** (только для absolute) — сетка из 5 кнопок (Top-Left, Top-Right, Center, Bottom-Left,
   Bottom-Right), живые показания beta/alpha, маленький визуализатор-канва с текущей точкой внутри
   калиброванного диапазона. По нажатию на угол сохраняется текущая ориентация. Когда собраны все 5 —
   автопереход в управление.
4. **Control** — полноэкранная зона управления, поделённая на левую и правую половины (ЛКМ/ПКМ),
   подсказка по жестам внизу.

## 5. Режимы управления курсором

Есть **два режима**, и ключевой принцип: **жест в обоих режимах одинаковый** (наводишь телефоном как
указкой), разница только в том, что absolute привязан к абсолютным углам через калибровку, а relative
работает по приращениям и не требует калибровки.

### 5.1. Absolute mode (с калибровкой)

Горизонталь курсора управляется углом `alpha` (поворот телефона влево-вправо), вертикаль — углом `beta`
(наклон вверх-вниз).

**Калибровка.** Пользователь наводит телефон на 4 угла рабочей области и центр, фиксируя по кнопке.
Точка Center сейчас собирается, но в расчётах **не используется** (можно либо задействовать, либо убрать
из UI — на усмотрение, но это известный «лишний» элемент).

Из 4 углов считаются границы диапазонов:

- верх (min beta) = среднее beta верхних двух точек;
- низ (max beta) = среднее beta нижних двух точек;
- лево (alpha_left) = среднее alpha левых двух точек;
- право (alpha_right) = среднее alpha правых двух точек.

**Разворачивание alpha (критично!).** `alpha` — это компасный азимут, он **зацикливается на 0/360**.
Если этого не учитывать, при уводе телефона за точку перехода через 0/360 доля по X скачком меняется
с ~1 на ~0 и курсор прыгает на противоположную сторону. Правильная обработка — привести правый край и
живое значение на **непрерывную ось вокруг центра калиброванного диапазона**:

1. если `alpha_right − alpha_left > 180` → `alpha_right −= 360`; если `< −180` → `alpha_right += 360`;
2. `center = (alpha_left + alpha_right) / 2`;
3. живое `alpha` подтянуть в пределы ±180 от center: пока `alpha − center > 180` вычитать 360; пока `< −180` прибавлять 360.

**Нормализация и маппинг.**

- доля по X: `frac_x = (alpha − alpha_left) / (alpha_right − alpha_left)`;
- доля по Y: `frac_y = (beta − min_beta) / (max_beta − min_beta)`;
- значения вне [0..1] означают, что телефон выведен за калиброванные края (полезно для отладки);
- горизонтальная цель раскладывается **на всю ширину виртуального рабочего стола**, чтобы указка
  пересекала все мониторы: `target_x = origin_x + frac_x · virtual_width`.

**Вертикаль по конкретному монитору (критично для мультимонитора!).** См. раздел 6.

**Сглаживание (EMA).** Финальная позиция = экспоненциальное скользящее среднее между целью и предыдущей
позицией: `pos = target · sf + last · (1 − sf)`, где `sf` — коэффициент сглаживания (со слайдера).
Чем меньше sf — тем плавнее и «ленивее» курсор, чем больше — тем «прямее».

**Динамический анти-джиттер.** Если включён: по ускорению с гравитацией считается модуль вектора;
`jitter_level = |модуль − 9.8|`. Если он превышает порог (резкое дрожание руки) — на этот кадр
коэффициент сглаживания принудительно занижается (сильнее сглаживаем), чтобы дрожь не передавалась в курсор.

### 5.2. Relative mode (air-mouse, без калибровки)

Тот же жест, что и в absolute (alpha = влево-вправо, beta = вверх-вниз), но курсор двигается по
**приращению наведения за кадр**, а не по абсолютной привязке. Поэтому калибровка не нужна.

Для каждого кадра:

- `da = alpha − предыдущий_alpha`, привести в диапазон [−180, 180] (alpha зацикливается!);
- `db = beta − предыдущий_beta`;
- функция формы для каждой дельты: если `|d| < deadzone` → 0 (гасит дрейф/дрожь); иначе
  `sign(d) · (|d| − deadzone)^exp · sensitivity`, где `sensitivity` — пикселей на градус,
  `exp ≥ 1` добавляет «мышиное ускорение» (точность на медленных движениях, скорость на размашистых);
- `dx = sign_x · форма(da)`, `dy = sign_y · форма(db)` (знаки конфигурируемы — оси могут зеркалиться);
- скорость низкочастотно фильтруется (EMA по dx/dy) для плавности;
- новая позиция = предыдущая + скорость, с клампом в границы виртуального стола.

Нюансы:

- нужны **два сэмпла**, чтобы взять дельту; «предыдущую ориентацию» сбрасывать при подключении и при
  смене режима, иначе первый кадр даст огромный скачок;
- `beta` опирается на гравитацию и не дрейфует; медленный дрейф `alpha` гасится мёртвой зоной;
- по умолчанию вертикаль обычно требует инверсии знака (целишься ниже → курсор ниже) — заложить
  настраиваемые знаки осей.

**Важный архитектурный урок:** relative **нельзя** делать по абсолютному углу наклона как по «скорости»
(тогда курсор улетает, потому что удержание телефона под любым углом = постоянная ненулевая скорость).
И не стоит уходить в проекции угловой скорости гироскопа (rotationRate) — на практике это даёт путаницу
осей и неинтуитивное «качание экраном». Дельты ориентации alpha/beta дают жест, идентичный absolute, —
это и есть правильное решение.

## 6. Мультимониторная геометрия (важный раздел)

Получить от ОС список мониторов (для каждого: позиция x/y и размеры) и посчитать
**виртуальный рабочий стол** — общий ограничивающий прямоугольник: origin = (min x, min y) по всем
мониторам, размер = (max(x+width) − min x, max(y+height) − min y).

**Проблема:** если мониторы разной высоты или со смещением по вертикали, общий прямоугольник
**не совпадает** с реальной областью экранов — остаются «мёртвые зоны» (углы, где монитора нет).
Линейный маппинг beta на всю высоту прямоугольника загоняет курсор в эти зоны, и ОС прибивает его к
ближайшему пикселю → «странный кламп», верх/низ не доезжают.

**Решение — вертикаль по монитору под курсором:**

1. горизонталь (`target_x`) считается по всей ширине виртуального стола (см. 5.1) — указка плавно
   пересекает оба монитора;
2. по `target_x` определить, над **каким реальным монитором** сейчас курсор (монитор, чей диапазон по X
   содержит target_x; если ни один — ближайший по центру);
3. горизонталь зажать в границы этого монитора;
4. **вертикаль растянуть на реальную высоту именно этого монитора**: `target_y = mon_y + frac_y · mon_height`,
   зажать в [mon_y, mon_y + mon_height − 1];
5. фолбэк (если раскладки нет): старый вариант — beta на всю высоту прямоугольника + кламп в общий бокс.

Так «верх/низ» всегда означают реальные края того экрана, на который наведена указка, без мёртвых зон.

**Нюанс (заложить, не баг):** на стыке мониторов разной высоты вертикальный «ноль» у экранов разный,
поэтому при пересечении стыка у верхней/нижней кромки курсор может слегка дёрнуться по Y. Это неизбежно
для разновысоких мониторов, приемлемо.

Раскладку читать **на старте** сервера. Вывести её в консоль.

## 7. Жесты на экране управления (тач)

Зона управления = вся область, поделённая на левую и правую половины. **Три жеста**, без устаревшего
«зажать обе половины»:

1. **Тап** (быстрое касание, палец почти не сдвинулся) → клик: левая половина — ЛКМ, правая — ПКМ.
2. **Удержание неподвижным пальцем** (дольше порога, например ~180 мс, без движения) → кнопка
   **прижимается и держится**; дальше пользователь водит телефоном (курсор едет от гироскопа) и так
   **выделяет файлы/текст**; на отпускании палец — кнопка отпускается. (Курсор во время удержания НЕ
   замораживается — это и есть drag.)
3. **Свайп пальцем по экрану** (палец сдвинулся больше порога, например ~8 px, раньше срабатывания
   удержания) → режим **скролла, как лента в TikTok**: вертикальное движение пальца крутит колесо
   (свайп вверх = лента вниз). Во время свайпа поток «движения» курсора подавляется, чтобы курсор не
   дёргался; величина прокрутки = дельта Y пальца между событиями touchmove.

Дизамбигуация: засекаем касание; если палец сдвинулся за порог до истечения таймера удержания — это
свайп-скролл; если палец стоит на месте и таймер истёк — прижимаем кнопку (удержание); если отпустили
быстро без сдвига — это тап-клик. Обрабатывать также touchcancel как отпускание.

Скорость скролла — отдельный конфиг на сервере. Направление скролла — конфигурируемо (возможна инверсия знака).

## 8. Список настраиваемых параметров (тюнинг)

Должны быть вынесены в явные константы наверху файлов.

**Сервер:**
- порт; предпочитаемый IP (override автоопределения); пути к сертификату/ключу;
- чувствительность скролла;
- сглаживание по умолчанию (если слайдер не прислал значение);
- анти-джиттер: порог дрожания, заниженный коэффициент сглаживания при дрожании;
- relative/air: чувствительность (px/градус), экспонента ускорения, мёртвая зона (градусы/кадр),
  сила сглаживания скорости, знаки осей X и Y;
- отладка: вкл/выкл подробного лога, интервал тротлинга лога.

**Клиент:**
- порог времени удержания (тап vs удержание);
- порог сдвига пальца (удержание vs свайп);
- диапазон/шаг слайдера сглаживания.

## 9. Логирование (для отладки калибровки/маппинга)

Подробный, но компактный и **тротлингованный** лог на сервере. Минимально полезные строки:

- по каждой точке калибровки: идентификатор и углы; по завершении — сводка границ (лево/право/верх/низ);
- по absolute-движению (раз в ~0.25с): входной alpha и его развёрнутое значение, границы лево/право,
  доли frac_x/frac_y, выбранный монитор, итоговая позиция курсора;
- по relative-движению: дельты da/db и итоговое смещение.

Лог должен быть лаконичным (экономия места в консоли), включаться/выключаться флагом.

## 10. Граничные случаи и уроки (обязательно учесть)

1. **alpha зацикливается на 0/360** — всегда разворачивать вокруг центра диапазона (absolute) и
   приводить дельту в [−180,180] (relative). Иначе прыжок курсора на противоположную сторону.
2. **alpha = компас** — дрейфует, зависит от магнитометра; absolute по горизонтали принципиально менее
   стабилен и требует калибровки. Это ограничение железа, не баг.
3. **Мультимонитор разной высоты** — общий бокс не прямоугольный, нужны мёртвые зоны → маппинг вертикали
   по конкретному монитору (раздел 6).
4. **relative только по дельтам ориентации**, не по абсолютному углу и не по проекциям rotationRate.
5. **Сброс «предыдущей ориентации»** при connect/смене режима — иначе скачок на первом кадре.
6. **iOS:** HTTPS + запрос разрешений строго внутри жеста-тапа.
7. **Скролл — свайп пальцем**, а не наклон и не «обе половины»; иначе теряется удержание-перетаскивание
   для выделения.
8. **Center-точка калибровки** в текущей математике не используется.
9. **Мягкая деградация** при недоступности любой из подсистем (ввод/мониторы/TLS).
10. **Единый порт** для страницы и сокета — чтобы сертификат принимался один раз.

---

## 11. Целевые платформы
- Телефон: iOS Safari (главный кейс по разрешениям) и Android Chrome.
- ПК-сервер: кроссплатформенно, но первичная цель — Windows (мультимонитор, виртуальный рабочий стол с
  отрицательными координатами). Linux требует отдельной проверки инъекции ввода под Wayland.

## 12. Запуск и TUI (выбор интерфейса + QR)

Демон должен быть удобным «запустил и забыл». Сценарий старта в терминале:

### 12.1. Интерактивный выбор сетевого интерфейса
При запуске показать **интерактивный список** доступных сетевых интерфейсов/IP (как `yarn upgrade
--interactive`):

- навигация **стрелками** ↑/↓, выбор **пробелом/Enter**, выход по Esc/Ctrl-C;
- по каждому пункту показать: имя интерфейса, IPv4, и пометку типа (LAN / VPN / virtual);
- **рекомендуемый** интерфейс (лучший по ранжированию из раздела 2.3) визуально помечен (например
  `(recommended)` и подсветка) и стоит курсором по умолчанию — Enter сразу выбирает его;
- если задан override-IP/`--ip` или передан флаг `--yes`/`--no-tui` — список пропускается, берётся
  рекомендуемый/заданный (для headless/автозапуска как сервис);
- если интерфейс ровно один — можно не показывать список, но вывести, какой выбран.

После выбора сервер биндится на `0.0.0.0` (слушаем все), но именно выбранный IP идёт в сертификат (SAN)
и в URL/QR для телефона.

### 12.2. QR для подключения
Сразу после выбора интерфейса и подъёма сервера — **напечатать в терминал QR-код** со ссылкой
`https://<выбранный_ip>:<порт>` (как `expo start` в react-native):

- QR рендерится ASCII/Unicode-блоками прямо в консоль (без графики);
- под QR — текстом продублировать URL и подсказку «отсканируй камерой телефона»;
- предупредить, что серт самоподписанный: на телефоне будет «небезопасно» → принять один раз;
- по желанию: горячая клавиша для пере-вывода QR и пункт «сменить интерфейс» без перезапуска.

### 12.3. Фоновый режим / трей
Для «крутится в фоне»:

- основной UX — **иконка в системном трее**: показывает статус (ждёт/подключён телефон), текущий
  IP/порт, пункт «показать QR», «сменить интерфейс», «Quit»;
- автозапуск при логине — опционально (per-OS), не обязателен для MVP;
- TUI-выбор интерфейса и QR показываются при первом/ручном запуске; в чисто фоновом режиме (autostart)
  TUI пропускается, берётся сохранённый/рекомендованный интерфейс.

## 13. Реализация на Rust (стек и нюансы)

Вся логика разделов 3–10 — обычная арифметика и переносится в код напрямую. Ниже платформенный стек.
Цель: **один статический бинарь**, дешёвый фоновый демон, без внешних рантаймов и без системного OpenSSL.

### 13.1. Структура
- Клиент остаётся единым HTML-файлом и **встраивается в бинарь** (`include_str!`) — отдаётся как
  HTTP-ответ. Итог: фактически один исполняемый файл.
- Сервер — один Rust-крейт (бинарный).

### 13.2. Стек крейтов
- async-рантайм: **tokio**;
- WebSocket + HTTP на одном порту: **tokio-tungstenite** для WS, обычный HTTP-ответ отдаём вручную
  (та же логика «есть `Upgrade: websocket` → WS, иначе отдать HTML»);
- TLS: **rustls** (чистый Rust, без системного OpenSSL);
- генерация самоподписанного сертификата: **rcgen** (SAN с выбранным IP + 127.0.0.1 + localhost),
  кэшировать на диск, перевыпускать при смене IP;
- управление мышью (курсор/клики/колесо): **enigo**;
- геометрия мониторов: **display-info** (кроссплатформенно) либо на Windows напрямую
  `EnumDisplayMonitors` через крейт **windows** для надёжности (важно из-за отрицательных координат
  виртуального стола);
- TUI-выбор интерфейса: **inquire** или **dialoguer** (готовый Select со стрелками, дефолтом и
  пометкой рекомендуемого — ровно «yarn --interactive»);
- QR в терминал: **qrcode** (рендер в Unicode-блоки) или **qr2term**;
- трей: **tray-icon** (+ нужен event-loop, напр. через winit/tao на десктопе);
- JSON: **serde** / **serde_json**; логи: **tracing** (с тем же тротлингом для потока движения).

### 13.3. Платформенные риски (заложить в план)
1. **enigo под Linux/Wayland** — инъекция ввода ограничена; на X11 ок. Проверить целевую среду заранее.
2. **Мультимонитор-крейты** местами недозрелые; на Windows предпочесть нативный
   `EnumDisplayMonitors`. Обязательно проверить виртуальный стол с отрицательными координатами.
3. **Трей требует нативный event-loop** на главном потоке — учесть в архитектуре (tokio в отдельном
   потоке/рантайме, GUI-loop на главном).
4. **Демонизация из коробки не даётся** — для фонового режима либо процесс+autostart, либо интеграция
   с systemd/launchd/Windows Service. Для MVP достаточно трей-иконки.

### 13.4. Альтернатива
**Go** закрывает ту же нишу (single-binary, кроссплатформа): `robotgo` (ввод), но тянет CGo и даёт более
тяжёлую кросс-сборку и бинарь. Rust предпочтительнее для «минимально дешёвого и аккуратного» демона;
Go — если важнее скорость написания.
